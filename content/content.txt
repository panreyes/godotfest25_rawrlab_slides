- TTBTT: Porting tips
	- Based on my experiencies, mostly 2D games.
	- Don't copy them now, I'll share the slides later :)
		- #1. Optimize the game for a mid-tier Android device.
		- #2. Don't use Godot 3.x for mid/large-sized projects
				- (often suffering from frame pacing issues).
		- #3. Only use stable versions of Godot.
		- #4. We have loads of RAM, but less CPU and GPU.
		- #5. Bake and preload as much as you can.
		- #6. Static art should not use shaders.
				- (Using modulate does not hit performance.)

- TTBTT: Porting tips
		- #7. An static background should just be a texture. Prerrender them to one texture.
		- #8. Use as few real-time lights as possible.
		- #9. Most Light2Ds can be replaced with Sprite2D with additive blending mode.
		- #10. Use RenderDoc to find out where you're wasting drawing calls.
		- #11. Texture/Mesh LOD biases for quick gains.
		- #12. Trigger all shaders & GPUParticles during load time.
		- #13. Huge PackedByteArrays in text format are extremely slow to read.
		- #14. Always export resources as binaries.
		- #15. For some optimizations you might need to go lower. C++ 
				- Luckily, Godot's code is pretty easy to read.
		- #16. Save materials as resources, don't keep them embedded in the scenes.

- TTBTT: Porting tips
		- #17. Always set up a fallback font.
				- Godot uses the first available system font, but not all platforms provide one.
 		- #18. Don't use WorkerThreadPools jobs for loading stuff.
				- Even the best juggler has a ball limit.
 		- #19. Disable Android frame pacing.
 		- #20. Don't save the game too often.
				- Save settings when leaving the settings menu.
				- Save collectibles after ending the level.
		- #21. You only need to load the game saves once.

- TTBTT: Porting tips
		- #22. Spend some time learning the debugging tools that the platform provides.
		- #23. Try tinkering with every setting in the Project Settings and Viewport properties.
		- #24. For single-player games, accept input from any available controllers.
